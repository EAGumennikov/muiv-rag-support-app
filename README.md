# Система управления знаниями поддержки

Учебный web-сервис управления знаниями службы поддержки пользователей с RAG-поиском, базой знаний, глоссарием, личными кабинетами, административным контуром и экспортом документов. Проект выполнен при прохождении преддипломной практики и подготовке ВКР в Московском Университете имени С.Ю. Витте.

Полное название web-сервиса: «Система управления знаниями службы поддержки пользователей с применением RAG-подхода для поиска и формирования решений».

Студент: Гуменников Евгений Андреевич.

Тема ВКР: «Совершенствование системы управления знаниями службы поддержки пользователей с применением RAG-подхода для поиска и формирования решений».

## Опубликованный ресурс

Демонстрационный контур опубликован по адресу:

[https://muiv.eagumennikov.synology.me](https://muiv.eagumennikov.synology.me)

Размещение выполнено в формате self-hosting на демонстрационном сервере DS920+ через Docker, Gunicorn, внешний runtime-каталог `workdir`, обратный прокси Synology и HTTPS-сертификат Let's Encrypt.

## Что реализовано

- подготовка HTML-корпуса базы знаний: нормализация в Markdown, чанкинг, embeddings и FAISS-индекс;
- RAG-поиск по базе знаний с формированием ответа через внешний LLM API и показом источников;
- публичный web-интерфейс на Flask и Jinja с 10 основными разделами;
- каталог базы знаний `/articles` с серверным поиском, фильтром по разделу, сортировкой и пагинацией;
- просмотр статьи, скачивание Markdown и экспорт статьи в `.docx`;
- SQL-слой на SQLAlchemy и SQLite с 14 таблицами;
- пользователи, роли, личные кабинеты, редакторский контур и административная панель;
- сохранение истории вопросов, RAG-ответов, источников, feedback и аудита;
- защита `/ask` авторизацией и дневным лимитом RAG-запросов;
- экспорт RAG-ответов в `.docx`, пользовательской истории и административных отчетов в `.xlsx`;
- отраслевой глоссарий и справочник команд первой диагностики для адаптации сотрудников;
- единый светлый интерфейс с адаптивной навигацией, skip-link, заметными focus-состояниями и `aria-live` для динамических блоков;
- Docker-контур размещения на DS920+ с подключением внешнего `muiv-rag-support-workdir` как volume.

## Структура репозитория

- `app.py` - точка входа Flask-приложения и маршруты web-сервиса.
- `db/` - SQLAlchemy Base, конфигурация подключения и ORM-модели.
- `services/` - сервисный слой RAG, статей, доступа, SQL-операций, markdown, feedback и экспорта.
- `scripts/` - подготовка корпуса, индексация, запуск, инициализация БД и служебные проверки.
- `templates/` - HTML-шаблоны публичных и закрытых страниц.
- `static/` - стили, JavaScript и статические ресурсы интерфейса.
- `config/` - примеры конфигурации без секретов.
- `docs/` - репозиторная документация по архитектуре, RAG, SQL, доступу, размещению, проверкам и этапам развития.

Основной вход в документацию: [docs/00_Обзор_документации.md](docs/00_Обзор_документации.md).

## Публичный контур

Верхнее меню содержит 10 основных разделов:

| Раздел | Маршрут | Назначение |

| RAG-поиск | `/search` | поиск ответа по базе знаний |
| База знаний | `/articles` | каталог статей подготовленного корпуса |
| Глоссарий | `/glossary` | отраслевые и ИТ / ИБ-сокращения |

Совместимые старые URL `/about-system`, `/about-kb` и `/about` перенаправляют пользователя на актуальный раздел `/about-project`.

## RAG-сценарий

Страница `/search` доступна публично, но активное формирование ответа выполняется через защищенный маршрут `/ask`. Запуск retrieval и обращение к внешней LLM доступны только авторизованным ролям `support_user`, `knowledge_editor` и `admin`.

Для авторизованного пользователя вопрос отправляется через AJAX / fetch без перезагрузки страницы. Интерфейс показывает индикатор обработки, ответ, карточки источников и ссылку на экспорт ответа в `.docx`, если ответ сохранен в SQL.

Дневной лимит RAG-запросов задается переменной окружения `RAG_DAILY_LIMIT`. Если переменная не указана, используется значение `10`. Для роли `admin` лимит не применяется.

## Роли и закрытые разделы

В приложении реализованы три роли:

- `support_user` - личный кабинет, история вопросов, сохраненные ответы, feedback и экспорт собственной истории;
- `knowledge_editor` - редакторский контур, анализ источников, обращений и контента базы знаний;
- `admin` - административная панель, пользователи, роли, общая история, feedback, аудит и административные выгрузки.

Пользователь видит только собственные вопросы, ответы и обращения. Администратор видит общесистемную историю, обращения пользователей, базовую статистику и последние события аудита.

## SQL и данные

В проекте подготовлены 14 таблиц:

`users`, `roles`, `user_roles`, `articles`, `article_files`, `glossary_terms`, `onboarding_pages`, `faq_items`, `feedback_messages`, `search_queries`, `rag_answers`, `rag_answer_sources`, `audit_logs`, `settings`.

По умолчанию SQLite-БД размещается во внешнем `workdir`, а путь задается через `RAG_APP_DATABASE_URL`. В Docker-контуре DS920+ используется путь:

```text
sqlite:////workdir/output/app/muiv_rag_support.sqlite3
```

Runtime-данные, индексы, embeddings, SQLite-БД, `.env`, API-ключи, временные выгрузки и локальные рабочие файлы не входят в репозиторий.

## Локальный запуск

Пример запуска в архитектуре `app + workdir`:

```bash
cd <каталог-приложения>
source .venv/bin/activate
pip install -r requirements.txt
python3 app.py
```

После запуска интерфейс будет доступен по адресу [http://127.0.0.1:5000](http://127.0.0.1:5000).

Для запуска с переменными окружения из внешнего рабочего каталога используется скрипт:

```bash
cd <каталог-приложения>
bash scripts/run_web.sh
```

Инициализация таблиц:

```bash
cd <каталог-приложения>
source .venv/bin/activate
python3 scripts/init_app_db.py
```

Инициализация демонстрационных ролей и пользователей:

```bash
cd <каталог-приложения>
source .venv/bin/activate
python3 scripts/init_demo_users.py
```

Пароли демонстрационных учетных записей задаются через переменные окружения `RAG_DEMO_ADMIN_PASSWORD`, `RAG_DEMO_EDITOR_PASSWORD` и `RAG_DEMO_SUPPORT_PASSWORD`.

## Docker-размещение на DS920+

На демонстрационном сервере используется схема:

- код приложения: `/volume1/MUIV_Diplom/04_development/muiv-rag-support-app`;
- рабочие данные RAG и секреты: `/volume1/MUIV_Diplom/04_development/muiv-rag-support-workdir`;
- контейнер: `muiv-rag-support`;
- порт локальной проверки: `5002`;
- публичный адрес: `https://muiv.eagumennikov.synology.me`.

Запуск:

```bash
sudo -n /usr/local/bin/docker compose up -d --build
```

Проверка контейнера:

```bash
sudo -n /usr/local/bin/docker ps
sudo -n /usr/local/bin/docker logs --tail 80 muiv-rag-support
```

В `compose.yaml` используется host network, потому что на DS920+ bridge-сеть Docker в текущей конфигурации не получает корректный DNS при сборке. PyTorch устанавливается в CPU-only варианте, чтобы не тянуть CUDA-зависимости. Размер рабочего образа после оптимизации составляет около 1.55 GB.

## Проверки

Зафиксированные проверки текущего состояния:

- публичные маршруты опубликованного HTTPS-контура возвращают HTTP 200;
- TLS-сертификат проходит проверку `openssl s_client` с `Verify return code: 0 (ok)`;
- адаптивность публичного контура проверена на desktop `1440x900`, tablet `768x1024` и mobile `390x844`;
- горизонтальное переполнение на ключевых публичных страницах не выявлено;
- через Flask test client проверялись публичные маршруты, закрытые контуры, роли, `/ask`, каталог статей, экспорт `.docx` и `.xlsx`;
- каталог базы знаний содержит 6546 статей и формирует 66 страниц при пагинации по 100 материалов.

Подробная трассировка требований приведена в [docs/10_Трассировка_требований.md](docs/10_Трассировка_требований.md).
