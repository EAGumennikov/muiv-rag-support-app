# Система управления знаниями поддержки

Учебный web-сервис управления знаниями службы поддержки пользователей с RAG-поиском, базой знаний, глоссарием, личными кабинетами, административным контуром и экспортом документов. Проект выполнен при прохождении преддипломной практики и подготовке ВКР в МУИВ.

Полное название web-сервиса: «Система управления знаниями службы поддержки пользователей с применением RAG-подхода для поиска и формирования решений».

## Контуры работы

### Репозиторий
Репозиторий содержит только код проекта, шаблоны конфигурации, веб-интерфейс и техническую документацию.

### Локальный рабочий каталог
Runtime-данные, виртуальное окружение, секреты и крупные артефакты хранятся вне репозитория в отдельном рабочем каталоге.

### Целевой демонстрационный контур
Для соответствия требованиям сетевой ресурс планируется разместить на бесплатном хостинге. Локальный `workdir` используется для разработки и проверки, а размещение на хостинге будет оформлено отдельным этапом с описанием адреса, порядка развертывания и ограничений демонстрационной среды.

## Важно
Тяжелые данные, секреты, индексы и runtime-артефакты в репозиторий не включаются.

## Веб-интерфейс

На текущем этапе веб-слой включает:
- публичные страницы `/`, `/search`, `/articles`, `/glossary`, `/diagnostics`, `/onboarding`, `/faq`, `/about-system`, `/about-kb`, `/testing`, `/feedback`
- хлебные крошки на всех пользовательских страницах
- защищенный маршрут `/ask` для обработки вопроса через реальный RAG-контур
- просмотр статьи `/article/<doc_id>`
- скачивание markdown `/download/<doc_id>`
- экспорт статьи в `.docx` через `/article/<doc_id>/export.docx`
- форму обратной связи

Веб-интерфейс использует существующий retrieval-контур и генерацию ответа через Yandex AI Studio по OpenAI-совместимому API. Страница `/search` остается публичной, но формирование RAG-ответа через `/ask` относится к защищенному контуру и доступно только после входа.

Для авторизованного пользователя `/search` отправляет вопрос через `fetch` без перезагрузки страницы, показывает индикатор формирования ответа, выводит ответ, источники и ссылку на экспорт ответа в `.docx`, если ответ был сохранен в SQL. Если JavaScript отключен, HTML-форма продолжает отправлять запрос стандартным POST-сценарием.

Дневной лимит RAG-запросов задается переменной окружения `RAG_DAILY_LIMIT`. Если переменная не указана, используется значение `10`. Лимит считается по текущей дате и SQL-истории пользователя до запуска retrieval и внешнего API. Для роли `admin` лимит не применяется.

На этапе 22 интерфейс приведен к сдержанной корпоративной стилистике: белый и очень светлый фон, синие акценты, темный текст, тонкие границы, единые кнопки, формы, карточки и таблицы. В шапке используется нейтральная декоративная SVG-пиктограмма атома. Официальные логотипы и внешние брендовые материалы не используются.

На этапе 23 добавлены два публичных справочных раздела:
- `/glossary` - отраслевой глоссарий с категориями, расшифровками, комментариями и поиском по странице;
- `/diagnostics` - команды первой диагностики для Windows, Linux, macOS и универсальных проверок.

Начальное наполнение хранится в сервисном слое: `services/glossary_service.py` и `services/diagnostics_service.py`. Такой вариант выбран как минимально безопасный, потому что текущая SQL-модель `GlossaryTerm` пока не содержит категории, расшифровки и примера использования. Дальнейшее развитие - перевод справочников в управляемый SQL-контур с CRUD в редакторском разделе.

Справочные разделы предназначены для адаптации новых сотрудников службы поддержки. Они содержат только нейтральные учебные формулировки и безопасные команды просмотра состояния; чувствительная внутренняя информация, закрытые ссылки, адреса систем и непубличные регламенты не используются.

## Роли и доступ

В приложении реализованы три авторизованные роли:
- `support_user` - рабочие пользовательские функции и личный кабинет
- `knowledge_editor` - рабочая зона редактора знаний и доступ к контентному контуру
- `admin` - полный доступ к административным разделам

Публичные страницы остаются доступными без входа.

Основные маршруты по ролям:
- публичный контур: `/`, `/search`, `/articles`, `/faq`, `/feedback` и другие открытые страницы
- авторизованный пользователь: `/cabinet`, `/cabinet/profile`, `/cabinet/history`, `/cabinet/saved-answers`, `/cabinet/feedback`, `/cabinet/help`
- редактор знаний и администратор: `/editor`, `/editor/content`, `/editor/sources`, `/editor/feedback`, `/editor/links`
- только администратор: `/admin`, `/admin/users`, `/admin/roles`, `/admin/feedback`, `/admin/feedback/<id>`, `/admin/history`, `/admin/content`, `/admin/audit`

Если пользователь авторизован, история вопросов, сформированные RAG-ответы и обращения обратной связи сохраняются с `user_id`. Публичные страницы без входа продолжают работать, но генерация RAG-ответа через `/ask` требует авторизации, чтобы защитить вычислительный контур перед размещением на бесплатном хостинге.

Пользователь видит только собственные вопросы, ответы и feedback. Администратор видит общесистемную историю, обращения всех пользователей, базовую статистику и последние события аудита.

## Экспорт документов

На этапе 20 добавлены файловые сценарии без сохранения выгрузок в репозиторий. Документы формируются в памяти через `BytesIO` и сразу отдаются пользователю.

Реализованы `.docx`:
- RAG-ответ по `/export/rag-answer/<rag_answer_id>.docx`;
- статья базы знаний по `/article/<doc_id>/export.docx`.

Реализованы `.xlsx`:
- моя история запросов по `/cabinet/history/export.xlsx`;
- административная выгрузка feedback по `/admin/export/feedback.xlsx`;
- административная выгрузка общей истории запросов по `/admin/export/history.xlsx`;
- административная статистика по `/admin/export/statistics.xlsx`.

Разграничение доступа:
- статья в `.docx` доступна публично, так как сама статья уже доступна через публичный маршрут;
- пользователь выгружает только свою историю и свои RAG-ответы;
- редактор знаний может выгружать RAG-ответы для анализа источников, но не получает административные `.xlsx`;
- административные `.xlsx` доступны только роли `admin`.

## SQL-слой и SQLite

В приложении добавлен демонстрационный SQL-слой на базе `SQLAlchemy`.

Что уже хранится в базе:
- сообщения формы обратной связи;
- история поисковых запросов;
- итоговые RAG-ответы;
- связанные источники ответа;
- базовый аудит событий.
- пользователи, роли и связи доступа;
- привязка истории запросов и feedback к пользователю, если обращение выполнено после входа.

На этапе 19 добавлены nullable-связи `user_id` для `search_queries` и `feedback_messages`. Ответ RAG связан с пользователем через таблицу `search_queries`, чтобы не дублировать связь в `rag_answers`.

По умолчанию используется SQLite во внешнем `workdir`:

`sqlite:///~/Projects/muiv-rag-support-workdir/output/app/muiv_rag_support.sqlite3`

Путь можно переопределить через переменную окружения `RAG_APP_DATABASE_URL`.

Для явной инициализации таблиц можно выполнить:

```bash
cd ~/Projects/muiv-rag-support-app
source .venv/bin/activate
python3 scripts/init_app_db.py
```

Для создания демонстрационных ролей и тестовых пользователей:

```bash
cd ~/Projects/muiv-rag-support-app
source .venv/bin/activate
python3 scripts/init_demo_users.py
```

Тестовые учетные записи по умолчанию:
- `admin / admin_demo_123`
- `editor / editor_demo_123`
- `support / support_demo_123`

Значения можно переопределить через переменные окружения:
- `RAG_DEMO_ADMIN_PASSWORD`
- `RAG_DEMO_EDITOR_PASSWORD`
- `RAG_DEMO_SUPPORT_PASSWORD`

## Соотношение repo и workdir

- `~/Projects/muiv-rag-support-app` - код, шаблоны, Flask-приложение, CLI-скрипты, документация
