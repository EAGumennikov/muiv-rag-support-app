# RAG-конвейер

RAG-модуль является прикладным поисковым ядром web-сервиса. Он отвечает за поиск релевантных фрагментов базы знаний и формирование ответа на основе найденного контекста.

## Общая схема

Пользовательский сценарий состоит из этапов:

1. Пользователь задает вопрос на странице `/search`.
2. Защищенный маршрут `/ask` получает вопрос от авторизованного пользователя.
3. Retrieval-слой находит релевантные чанки.
4. Answer pipeline очищает и группирует найденные фрагменты.
5. Формируется prompt с вопросом и контекстом.
6. Внешняя LLM формирует итоговый ответ.
7. Система сохраняет историю запроса, ответ и источники.
8. Пользователь видит ответ и карточки источников.

## Разделение retrieval и generation

Retrieval и generation рассматриваются как разные слои.

Retrieval отвечает за:

- загрузку индекса;
- поиск ближайших фрагментов;
- отбор релевантных чанков;
- передачу metadata источников.

Generation отвечает за:

- сбор контекста;
- формирование prompt;
- обращение к внешней модели;
- получение ответа.

Такое разделение позволяет независимо улучшать качество поиска и качество текста ответа.

## Источники ответа

В пользовательской выдаче источники отображаются человекочитаемо:

- название статьи;
- breadcrumbs;
- секция документа;
- краткий фрагмент;
- ссылка на статью;
- ссылка на оригинал при наличии.

Технические поля используются только внутри приложения и не являются основным пользовательским названием источника.

## Интеграция с LLM

Генеративный слой подключен через OpenAI-совместимый API Yandex AI Studio. Секреты, ключи и параметры окружения не хранятся в репозитории.

Вход модели ограничивается найденными фрагментами, а не полным корпусом. Это уменьшает объем контекста и помогает привязать ответ к конкретным материалам.

## Оптимизация стоимости и ответа

Для снижения стоимости и повышения прикладной ценности ответа были применены:

- сокращение числа чанков, передаваемых в модель;
- ограничение длины контекста;
- ограничение объема ответа;
- упрощение prompt;
- приоритет специализированных секций над общими обзорными блоками.

## Сохранение истории

После обработки запроса в SQL сохраняются:

- текст вопроса;
- debug-метрики retrieval;
- текст ответа;
- список источников;
- связь с пользователем, если запрос выполнен после входа.

Эта история используется в личном кабинете, административном контуре и экспортных сценариях.

## Защита опубликованного контура

Страница `/search` остается публичной, но выполнение RAG-запроса через `/ask` доступно только после входа. Это защищает retrieval-слой и внешний LLM API в опубликованном demo-контуре.

Для ролей `support_user` и `knowledge_editor` применяется дневной лимит запросов через `RAG_DAILY_LIMIT`. Для роли `admin` лимит не применяется, чтобы администратор мог выполнять демонстрационную проверку сценариев.
