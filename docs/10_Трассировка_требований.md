# Соответствие требованиям сетевого ресурса

Документ фиксирует, как текущая реализация закрывает основные требования к сетевому ресурсу для практики и ВКР.

## Технологическая основа

Требование: использовать Python с Flask или аналогичный web-фреймворк.

Реализация:

- приложение написано на Python;
- web-слой построен на Flask;
- шаблоны реализованы через Jinja;
- прикладная логика вынесена в сервисы.

## Публичный контур

Требование: наличие 10+ публичных страниц, меню и хлебных крошек.

Реализация:

- главная страница;
- поиск;
- каталог статей;
- глоссарий;
- команды первой диагностики;
- onboarding;
- FAQ;
- о системе;
- о базе знаний;
- тестирование;
- обратная связь;
- страницы статей.

Дополнительно на этапе визуальной полировки:

- интерфейс приведен к единой светлой стилистике с синими акцентами;
- шапка содержит короткое название сервиса, подпись и нейтральную атомную пиктограмму;
- публичная навигация, вход и закрытые разделы визуально разделены;
- карточки, формы, кнопки и таблицы унифицированы;
- официальный логотип и внешние брендовые материалы не используются.

Дополнительно на этапе развития справочных разделов:

- `/glossary` превращен в отраслевой глоссарий с категориями, расшифровками и поиском;
- `/diagnostics` добавлен как публичный справочник команд первой диагностики;
- онбординг связывает RAG-поиск, базу знаний, глоссарий и диагностические команды в единый сценарий адаптации;
- справочные материалы не содержат чувствительной внутренней информации, закрытых адресов систем и непубличных регламентов.

## SQL-СУБД и таблицы

Требование: SQL-СУБД и не менее 10 таблиц.

Реализация:

- используется SQLAlchemy;
- локально используется SQLite;
- подготовлено 14 таблиц;
- хранятся пользователи, роли, feedback, история запросов, ответы RAG, источники и аудит.

## Роли и кабинеты

Требование: не менее 3 ролей пользователей, личные кабинеты и административная панель.

Реализация:

- `support_user`;
- `knowledge_editor`;
- `admin`;
- личный кабинет содержит 5+ разделов;
- редакторский контур выделен отдельно;
- административный контур содержит 5+ страниц.

Визуальное оформление закрытого контура усилено:

- записи истории, feedback, аудита и статистики отображаются в таблицах;
- статусы feedback выделяются бейджами;
- быстрые действия и пустые состояния оформлены единообразно;
- адаптивность сохранена для узких экранов.

## Обратная связь

Требование: форма обратной связи.

Реализация:

- публичная форма `/feedback`;
- сохранение в SQL;
- привязка к пользователю при наличии входа;
- просмотр в админке;
- изменение статуса обращения.

## Адаптация сотрудников

Требование: web-сервис должен быть полезен для практической работы и адаптации пользователей.

Реализация:

- создан отраслевой глоссарий атомного и ИТ / ИБ-контекста;
- добавлены команды первой диагностики по Windows, Linux, macOS и универсальным проверкам;
- в справочниках есть фильтры и поиск по странице;
- разделы доступны публично и связаны с онбордингом;
- редакторский контур показывает обзор количества терминов и команд.

## Файловые сценарии и экспорт

Требование: доступ к файловой системе и формирование документов.

Реализация:

- чтение подготовленных Markdown-статей;
- скачивание Markdown;
- экспорт статьи в `.docx`;
- экспорт RAG-ответа в `.docx`;
- экспорт пользовательской истории в `.xlsx`;
- экспорт feedback, общей истории и статистики в `.xlsx`.

## RAG-функциональность

Требование по теме ВКР: использовать RAG-подход для поиска и формирования решений.

Реализация:

- подготовлен корпус знаний;
- выполнен чанкинг;
- построены embeddings;
- построен FAISS-индекс;
- реализован retrieval;
- реализовано формирование ответа;
- в выдаче отображаются источники;
- история RAG-взаимодействий сохраняется в SQL.

Дополнительно усилен пользовательский сценарий:

- `/search` доступен публично как информационная страница поиска;
- активная генерация ответа через `/ask` доступна только авторизованным ролям;
- вопрос отправляется через AJAX / fetch без перезагрузки страницы;
- интерфейс показывает индикатор обработки, ошибки и карточки источников на текущей странице;
- добавлен дневной лимит RAG-запросов через `RAG_DAILY_LIMIT`, чтобы защитить demo-контур перед размещением.

Интерактивный блок `/search` визуально оформлен как основной рабочий сценарий: форма вопроса, подсказка о защищенном контуре, дневной лимит, индикатор обработки, ошибки, ответ, источники и ссылка на DOCX-экспорт используют общий стиль интерфейса.

## Размещение

Требование: разместить ресурс на бесплатном хостинге.

Текущий статус:

- требование зафиксировано как следующий обязательный этап;
- в документации выделена статья [Размещение на бесплатном хостинге](09_Размещение_и_запуск.md);
- `/ask` защищен авторизацией и дневным лимитом перед будущей публикацией;
- после выбора площадки нужно указать адрес, порядок развертывания и ограничения.

## Уточнение целевой аудитории после этапа 27

Публичный контур ориентирован на две основные аудитории:

- новые сотрудники службы поддержки - используют сервис для адаптационной подготовки в период испытательного срока, изучают материалы по направлениям, работают с RAG-поиском, источниками, глоссарием и диагностическими командами;
- действующие сотрудники поддержки - используют сервис как рабочий инструмент поиска решений по редким, новым и нетиповым кейсам.

Сопровождающие роли:

- руководитель или наставник использует материалы сервиса для подготовки сотрудника и ручного составления итоговой проверки знаний;
- редактор знаний и администратор поддерживают актуальность базы знаний, feedback, статей и справочных разделов.

Это уточнение усиливает требования к прикладной ценности публичных страниц: главная страница, `/onboarding`, `/testing`, `/search`, `/about-project` и `/faq` должны объяснять пользовательские сценарии, а не только техническое устройство прототипа.

## Инвентаризация SQL-слоя

| Таблица | Назначение | Ключевые связи |
|---|---|---|
| `users` | учетные записи | связана с `roles`, `search_queries`, `feedback_messages` |
| `roles` | справочник ролей | связана с `users` через `user_roles` |
| `user_roles` | назначение ролей | many-to-many между пользователями и ролями |
| `articles` | карточки статей БЗ | связана с `article_files`, `rag_answer_sources` |
| `article_files` | файлы статей | связана с `articles` |
| `glossary_terms` | базовая SQL-заготовка глоссария | требует расширения перед CRUD |
| `onboarding_pages` | страницы адаптации | справочный контент |
| `faq_items` | FAQ | справочный контент |
| `feedback_messages` | обращения пользователей | nullable-связь с `users` |
| `search_queries` | история вопросов | nullable-связь с `users`, связь с `rag_answers` |
| `rag_answers` | сохраненные ответы | связь с `search_queries` |
| `rag_answer_sources` | источники ответа | связь с `rag_answers` и статьями |
| `audit_logs` | журнал событий | фиксирует действия пользователей и системы |
| `settings` | прикладные настройки | служебная таблица |

Глоссарий и команды первой диагностики на текущем этапе демонстрационно хранятся в сервисном слое (`services/glossary_service.py`, `services/diagnostics_service.py`). Это решение снижает риск миграций перед demo-хостингом; дальнейший шаг - перенос в управляемый SQL-контур с CRUD для редактора знаний.

## Репозиторий и runtime-артефакты

Ревизия файлового состава:

- отслеживаемые git-файлы: около `484K`;
- рабочая папка без `.git`: около `934M`;
- `.venv`: около `933M`;
- рабочая папка без `.git` и `.venv`: около `1.0M`;
- неотслеживаемые `.env`, SQLite-БД, индексы, embeddings, логи, `.docx`, `.xlsx` в рабочей папке не обнаружены.

В репозитории должны храниться код, шаблоны, стили, скрипты, конфигурационные примеры и документация. Runtime-данные, секреты, локальная БД, индексы, embeddings и пользовательские выгрузки должны оставаться во внешнем рабочем контуре или передаваться через настройки выбранного хостинга.
