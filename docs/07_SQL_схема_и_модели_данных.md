# SQL-схема и прикладные данные

SQL-слой реализован на SQLAlchemy. Для локальной демонстрации используется SQLite, путь к базе задается через `RAG_APP_DATABASE_URL`.

## Назначение SQL-слоя

SQL-БД хранит прикладные данные web-сервиса:

- пользователей;
- роли;
- связи пользователей и ролей;
- сообщения обратной связи;
- историю поисковых запросов;
- RAG-ответы;
- источники ответов;
- аудит событий;
- справочные сущности для дальнейшего развития.

## Таблицы

В проекте подготовлены 14 таблиц:

- `users`;
- `roles`;
- `user_roles`;
- `articles`;
- `article_files`;
- `glossary_terms`;
- `onboarding_pages`;
- `faq_items`;
- `feedback_messages`;
- `search_queries`;
- `rag_answers`;
- `rag_answer_sources`;
- `audit_logs`;
- `settings`.

## Инвентаризация связей

Ключевые связи SQL-слоя:

- `users` связывается с `roles` через `user_roles`;
- `search_queries.user_id` связывает историю вопросов с авторизованным пользователем;
- `rag_answers.search_query_id` связывает итоговый ответ с исходным вопросом;
- `rag_answer_sources.rag_answer_id` хранит список источников конкретного ответа;
- `feedback_messages.user_id` связывает обращение обратной связи с пользователем, если оно отправлено после входа;
- `audit_logs` фиксирует события входа, выхода, feedback, RAG-взаимодействий и изменения статусов.

В RAG-истории участвуют `search_queries`, `rag_answers`, `rag_answer_sources` и при наличии связи `users`. В feedback-контуре участвуют `feedback_messages`, `users` и `audit_logs`. В разграничении доступа участвуют `users`, `roles` и `user_roles`.

## Пользователи и роли

`users` хранит учетные записи. `roles` хранит справочник ролей. `user_roles` связывает пользователей и роли по схеме многие-ко-многим.

Демонстрационные роли:

- `support_user`;
- `knowledge_editor`;
- `admin`.

## История RAG

`search_queries` хранит:

- текст вопроса;
- канал запроса;
- количество найденных фрагментов;
- количество использованных фрагментов;
- debug-данные;
- дату создания;
- nullable `user_id`.

`rag_answers` хранит итоговый ответ и связан с `search_queries` через `search_query_id`.

`rag_answer_sources` хранит источники, использованные при формировании конкретного ответа.

## Feedback

`feedback_messages` хранит:

- имя;
- email;
- тему;
- текст сообщения;
- статус;
- дату создания;
- nullable `user_id`.

Статусы обращений:

- `new`;
- `in_progress`;
- `resolved`;
- `rejected`.

## Аудит

`audit_logs` фиксирует базовые события:

- вход пользователя;
- выход пользователя;
- сохранение feedback;
- сохранение RAG-взаимодействия;
- изменение статуса feedback.

## Совместимость старых данных

Поля `user_id` в `search_queries` и `feedback_messages` nullable. Это позволяет сохранять совместимость с уже накопленными публичными записями без пользователя.

Для существующей SQLite-БД предусмотрено аккуратное добавление новых nullable-колонок при инициализации приложения.

## Глоссарий и диагностический справочник

В SQL-схеме есть таблица `glossary_terms`, но текущее демонстрационное наполнение отраслевого глоссария хранится в `services/glossary_service.py`. Это сделано потому, что текущая модель содержит только `term`, `definition` и `source_url`, а интерфейсу нужны категория, расшифровка, комментарий и пример использования.

Справочник команд первой диагностики хранится в `services/diagnostics_service.py` и пока не имеет отдельной SQL-таблицы.

Возможное развитие SQL-слоя после текущей версии:

- расширить модель глоссария полями категории, расшифровки, комментария и примера;
- использовать уникальность по связке `term + category`;
- добавить таблицу диагностических команд или универсальный справочник материалов адаптации;
- реализовать CRUD в редакторском контуре.
